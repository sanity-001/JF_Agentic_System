# 技能与插件系统 —— 深度源码解析

> **关键文件**: `skills/loader.py`、`skills/registry.py`、`plugins/loader.py`、`commands/registry.py`  
> **兼容性**: 兼容 `anthropics/skills` 和 `claude-code/plugins`  
> **核心概念**: Skills = 按需加载的知识，Plugins = 可组合的扩展包

## 1. Skills 系统 —— 按需加载的知识

### 1.1 什么是 Skill？

Skill 是一个 Markdown 文件（`SKILL.md`），包含特定领域的专家指导。模型在需要时加载，不需要时不占用上下文。

```markdown
---
name: my-skill
description: Expert guidance for my specific domain
---

# My Skill

## When to use
Use when the user asks about [your domain].

## Workflow
1. Step one
2. Step two
...
```

### 1.2 多级技能目录

OpenHarness 从多个位置发现技能，按来源分类：

```
技能加载优先级（后注册的覆盖先注册的）:

1. Bundled（内置）        ← 随 OpenHarness 发布的基础技能
   └─ openharness/skills/bundled/

2. User（用户级）         ← 用户个人技能
   └─ ~/.openharness/skills/<name>/SKILL.md
   └─ ~/.claude/skills/<name>/SKILL.md        ← 兼容 Claude Code
   └─ ~/.agents/skills/<name>/SKILL.md        ← 兼容其他 Agent 工具

3. Project（项目级）      ← 项目特定技能
   └─ <project>/.openharness/skills/<name>/SKILL.md
   └─ <project>/.claude/skills/<name>/SKILL.md
   └─ <project>/.agents/skills/<name>/SKILL.md

4. Plugin（插件级）       ← 插件携带的技能
   └─ <plugin>/skills/<name>/SKILL.md
```

### 1.3 load_skill_registry() 源码解析

```python
# skills/loader.py 第 42-75 行
def load_skill_registry(
    cwd: str | Path | None = None,
    *,
    extra_skill_dirs=None,
    extra_plugin_roots=None,
    settings=None,
) -> SkillRegistry:
    registry = SkillRegistry()
    
    # 1. 加载内置技能（bundled skills）
    for skill in get_bundled_skills():
        registry.register(skill)
    
    # 2. 加载用户级技能（~/.openharness/skills/ 及兼容目录）
    for skill in load_user_skills():
        registry.register(skill)
    
    # 3. 加载额外指定的技能目录
    for skill in load_skills_from_dirs(extra_skill_dirs, source="user"):
        registry.register(skill)
    
    # 4. 如果允许项目技能，从当前项目加载
    if cwd is not None and settings.allow_project_skills:
        project_dirs = discover_project_skill_dirs(cwd, ...)
        for skill in load_skills_from_dirs(project_dirs, source="project"):
            registry.register(skill)
    
    # 5. 从启用的插件中加载技能
    if cwd is not None:
        for plugin in load_plugins(settings, cwd):
            if plugin.enabled:
                for skill in plugin.skills:
                    registry.register(skill)
    
    return registry
```

### 1.4 SkillRegistry —— 注册与查询

```python
class SkillRegistry:
    def register(self, skill: SkillDefinition) -> None:
        """注册技能：按 name 存储，后注册的覆盖先注册的"""
    
    def list_skills(self) -> list[SkillDefinition]:
        """列出所有已注册技能"""
    
    def get(self, name: str) -> SkillDefinition | None:
        """按名称查找技能"""
    
    def get_skill_content(self, name: str) -> str | None:
        """获取技能的实际 Markdown 内容（用于拼入系统提示词）"""
```

### 1.5 Skill 的类型定义

```python
@dataclass
class SkillDefinition:
    name: str             # 技能名
    description: str      # 简短描述
    content: str          # Markdown 正文
    source: str           # "bundled" | "user" | "project" | "plugin"
    path: Path | None     # 磁盘路径
    user_invocable: bool  # 是否可作为斜杠命令直接调用
    enabled: bool = True  # 是否启用
```

### 1.6 项目技能的安全开关

项目技能默认启用，但在不信任的仓库中可以关闭：

```bash
oh config set allow_project_skills false
```

对应代码：

```python
# loader.py 第 59 行
if cwd is not None and getattr(resolved_settings, "allow_project_skills", True):
    project_dirs = discover_project_skill_dirs(...)
```

## 2. Plugins 系统 —— 可组合的扩展包

### 2.1 插件结构

一个标准插件的目录结构：

```
my-plugin/
└── .claude-plugin/
    ├── plugin.json          # 插件清单（必需）
    ├── commands/            # 斜杠命令
    │   └── my-command.md
    ├── hooks/               # 生命周期钩子
    │   └── hooks.json
    ├── agents/              # 自定义 Agent
    │   └── my-agent.md
    └── skills/              # 技能
        └── my-skill/
            └── SKILL.md
```

### 2.2 plugin.json 清单格式

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "My custom plugin",
  "author": "...",
  "homepage": "..."
}
```

### 2.3 插件发现路径

```
1. Bundled 插件           ← 随 OpenHarness 发布
2. User 插件              ← ~/.openharness/plugins/
3. Project 插件           ← <project>/.openharness/plugins/
4. 从源安装的插件          ← oh plugin install <source>
```

### 2.4 插件管理 CLI

```bash
oh plugin list              # 列出已安装
oh plugin install <source>  # 安装
oh plugin uninstall <name>  # 卸载
oh plugin enable <name>     # 启用
# (disable 通过 enable=false 的 config set)
```

### 2.5 已验证兼容的官方插件

| 插件 | 类型 | 功能 |
|------|------|------|
| `commit-commands` | Commands | Git commit / push / PR 工作流 |
| `security-guidance` | Hooks | 文件编辑安全检查警告 |
| `hookify` | Commands + Agents | 自定义行为钩子创建 |
| `feature-dev` | Commands | 功能开发工作流 |
| `code-review` | Agents | 多 Agent PR Review |
| `pr-review-toolkit` | Agents | 专项 PR Review Agent |

## 3. Commands 系统 —— 斜杠命令

### 3.1 命令架构

```
用户输入 "/commit"
  │
  ▼
┌───────────────────────────────────────┐
│  CommandRegistry.lookup("/commit")    │
│    ├─ 精确匹配 command name            │
│    ├─ 解析参数 "/commit -m 'fix bug'"  │
│    └─ 返回 (CommandDefinition, args)  │
│                                       │
│  ⚡ 如果是远程可调用命令：              │
│    → ohmo gateway 可以代理执行          │
└───────────────────────────────────────┘
```

### 3.2 命令定义

```python
@dataclass
class CommandDefinition:
    name: str                       # "commit"
    description: str                # "Create a well-structured git commit"
    handler: Callable               # 实际执行函数
    remote_invocable: bool = False  # 是否可通过 ohmo 远程调用
    remote_admin_opt_in: bool = False
```

### 3.3 命令分两类

| 类型 | 示例 | 特点 |
|------|------|------|
| **Read-Only** | `/help`, `/status`, `/cost`, `/skills` | 只检查状态，不修改文件 |
| **Stateful（有状态）** | `/commit`, `/clear`, `/plan`, `/plugin` | 修改本地状态或文件 |

`cli.py` 在 dry-run 模式下会显示命令的行为类别（第 124-201 行）。

## 4. Skills vs Commands vs Tools 的对比

| 维度 | Skills | Commands | Tools |
|------|--------|----------|-------|
| **本质** | 知识（Markdown） | 用户交互入口 | 模型可调用的能力 |
| **触发者** | 模型决定加载 | 用户输入 `/xxx` | 模型在 tool_use 中请求 |
| **格式** | SKILL.md | Python 函数 + 注册 | BaseTool 子类 |
| **加载时机** | 上下文组装时 | 用户输入时 | 始终在 tool schema 中 |
| **上下文占用** | 较大（完整的指令文本） | 小（仅命令名和描述） | 较小（JSON Schema） |
| **扩展性** | 写 Markdown 即可 | 写函数 + 注册 | 写类 + 注册 |

## 5. Dry-Run 中的技能/命令预览

OpenHarness 的 `--dry-run` 做了一个巧妙的智能匹配：

```python
# cli.py _recommend_preview_candidates()
# 将用户 prompt 分词后，与技能名/描述/内容匹配打分

例如 prompt: "I need to fix a bug in authentication"

1. 分词: ["fix", "bug", "authentication"]
2. 与每个 Skill 的名字+描述比对，计算匹配分
3. 返回得分最高的 5 个技能、8 个工具、5 个命令

输出:
  Likely Matches
  - skills:
    - debug (score=16) [fix, bug]
    - security-review (score=8) [authentication]
  - tools:
    - Read (score=8) [fix]
    - Bash (score=6) [fix]
```

这让用户在运行前就知道哪些技能和工具可能会被触发。

## 6. 插件系统的好处

1. **社区生态**：兼容 `anthropics/skills` 和 `claude-code/plugins`，可以复用已有的社区资源
2. **关注点分离**：核心引擎保持简洁，特殊功能通过插件注入
3. **安全边界**：插件可以被禁用，不信任的插件不会运行
4. **独立更新**：插件可以独立于 OpenHarness 核心更新
