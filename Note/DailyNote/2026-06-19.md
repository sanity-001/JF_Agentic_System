## 二、现有问题诊断

### 2.1 旧 Skill 的核心缺陷

文件：`OpenHarness/.claude/skills/experiment-control/SKILL.md`

| 问题 | 说明 |
|------|------|
| **无状态感知** | 不知道水冷当前是否在运行，无法判断指令是否合理 |
| **无法编排工作流** | 只能单步执行，不能"降温→稳定→采集→分析" |
| **无安全联锁** | 采集前不检查水冷状态和温度 |
| **无数据解读** | 只返回原始 JSON |
| **被动翻译** | 本质是"自然语言→curl"的短语手册 |

### 2.2 根本原因

**Skill 里不应该有 curl 命令。** Skill 应描述领域知识和工作流，执行应由 Tool 完成。旧方案混淆了"脑子"和"手脚"。

## 三、架构设计

### 3.1 总体架构

```
OpenHarness Agent Runtime（不动）
  └── Experiment Control Plugin（新建）
       ├── Tools (Python, ~36 个) — 调用 FastAPI
       ├── Skills (Markdown, 3 个) — 领域知识
       └── Agents (safety-watcher) — 后台监控
            │
            ▼ HTTP
       JF_Control_System FastAPI（不动）
            │
            ▼
       物理硬件
```

### 3.2 设计原则

| 原则 | 说明 |
|------|------|
| **不改后端** | 仅新增 `/mode` 接口（6 行） |
| **Tool 做手脚，Skill 做脑子** | Tool=执行+校验+翻译，Skill=知识+工作流+规则 |
| **安全联锁在 Tool 层硬编码** | 不可绕过 Agent 推理 |
| **混合粒度** | 常规用复合 Tool，调试用细粒度 Tool |
| **Tool 不做业务逻辑重复** | 校验在 Tool 层作为纵深防御，不替代后端校验 |

### 3.3 连接方式确认

- **正常连接**：`detector_load_config("path/to/config.config")` — 加载本地配置文件自动连接
- **备用连接**：`detector_connect(hostname, params)` — 不用配置文件时使用
- **采集模式**：`detector_set_mode("baseline"|"signal")` — 区分基线/信号采集

### 3.4 系统启动方式

```bash
conda run -n slsdet9 python start.py
```
- 直接调用项目自带的 `start.py`，不重复实现
- `conda run -n slsdet9` 自动激活 conda 环境
- `start_new_session=True` 创建独立进程组，`os.killpg` 一键终止
- 目标系统：Linux（不用 Windows 的 taskkill）

## 七、领域知识修正

### 7.1 X 光机联锁
- 基线采集不需要 X 光，不询问
- 基线→信号之间强制暂停，询问用户确认 X 光已开启
- 纯信号采集（已有基线）在采集前询问
- 未来如果接入 X 光控制，此步骤可自动化

### 7.2 流量报警阈值
- **正确值**：< 2 L/min（不是 5 L/min）
- 低于 2 L/min → 报警"可能管道堵塞"
