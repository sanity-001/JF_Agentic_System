# 2026-06-23 前端-Agent 状态同步修复

> 今日核心：消灭 Agent 和前端状态不同步的各种问题

---

## 一、conda 环境激活修复

### 1.1 VIRTUAL_ENV 冲突
**问题**：Agent 通过 `uv run oh` 启动，`VIRTUAL_ENV` 被子进程继承。`conda activate` 不清理这个变量，导致 `python` 指向 OpenHarness 的 .venv（Python 3.13）而非 conda 的 slsdet9。

**修复**：`start.sh` 开头 `unset VIRTUAL_ENV`，并用 `$CONDA_PREFIX/bin/python` 绝对路径。

### 1.2 start.sh 设计
- source conda.sh → conda activate slsdet9 → 启动 backend + frontend
- trap cleanup EXIT 确保 Ctrl+C 时清理子进程
- `system_tools.py` 改为 `bash start.sh`，删除手动 PATH 拼接

## 二、detector_run_acquisition 重构

**问题**：组合工具内含 load_config → set_mode → set_params → acquire，load_config 会把用户通过 detector_set_param 微调的参数覆盖成默认值。

**修复**：砍掉配置加载、模式设置、参数设置。detector_run_acquisition 只做：安全联锁 → 启动采集 → 等待完成 → 返回路径。

| 步骤 | 工具 |
|------|------|
| 加载配置 | detector_load_config |
| 查看参数 | detector_get_params |
| 修改参数 | detector_set_param |
| 设置模式 | detector_set_mode |
| 开始采集 | detector_run_acquisition(mode) |

各管各的，不再互相覆盖。

## 三、基线状态存储（三轮迭代）

### 3.1 第一轮：context.metadata（失败）
`context.metadata["has_baseline"] = True` — 但 OpenHarness 每次 tool 调用创建全新的 ToolExecutionContext（从 context.tool_metadata 展开副本），写入后即丢弃。下一个 tool 拿到空副本。

### 3.2 第二轮：磁盘 JSON 文件（可用但不优雅）
`.baseline_state.json` 写入 repo 根目录，但和前端逻辑不一致。

### 3.3 第三轮：后端 API（最终方案）
**根因发现**：前端能区分基线/信号是因为每次采集后调 `processVisual()`，后端在 `self._baselines` 内存字典里存着基线数据。前端通过响应中的 `baseline` 字段判断。Agent 另起炉灶自己追踪当然不行。

**最终方案**：
- 新增 `GET /api/detector/baseline/status` → `{"has_baseline": bool}`
- detector_run_acquisition signal 检查时调这个接口
- Agent 和前端共享同一个真相来源

## 四、前端 WebSocket 自动响应 Agent 操作

### 4.1 自动启动轮询
**问题**：`_startPolling()` 只在用户点前端按钮时调用。Agent load_config 后前端收不到参数和温度。

**修复**：WebSocket 检测到 `connected: false→true` 时自动启动轮询。

### 4.2 自动清空状态
**问题**：Agent shutdown 后前端参数、温度、图像残留。

**修复**：WebSocket 检测到 `connected: true→false` 时清空 params/temperatures/visualData/hasBaseline/acqMode。

## 六、架构原则总结

1. **前端和 Agent 共享后端作为唯一真相来源** — 所有状态查询走 API
2. **Agent 和前端独立进程，靠 WebSocket 同步** — 前端检测 WebSocket 状态变化自动响应
3. **工具职责单一** — 采集只管采集，配置只管配置，参数只管参数
