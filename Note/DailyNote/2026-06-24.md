# 2026-06-24 前端采集图像与进度条同步修复

> 今日核心：Agent 触发的采集在前端无法显示图像和进度条的根因排查与修复。经历了多轮迭代，最终定位到前端检测机制的多个竞态问题。

---

## 一、基线检测失败（后端薄包装漏属性）

### 现象
Agent 采完基线后，signal 采集时报"尚未采集基线"。

### 根因
后端 `self._baselines` 只在 `process_acquisition_visual()` 时写入。Agent 的 `detector_run_acquisition` 调了 `POST /visual/process`，但 `GET /baseline/status` 接口里 `_detector.baseline` 调用失败——`service.py` 薄包装没有转发 `baseline` 属性，抛 `AttributeError` → 500 → Agent 拿到 `{"detail": "..."}`（没有 `has_baseline` 字段）→ 判断为 false。

### 修复
`service.py` 添加 `baseline` 属性转发：
```python
@property
def baseline(self):
    return self._original.baseline
```

## 二、前端进度条与图像不显示（Agent 触发时）

### 现象
- 前端手动采集：进度条正常、图像正常
- Agent 触发采集：进度条不出现、图像不显示
- raw 文件正常保存

### 架构对比

**手动采集流程：**
```
按钮 → progress.value.acquiring = true（同步）→ startLocalProgress() → startAcquisition()
WebSocket 500ms 检测到 acquiring:false → progress.acquiring=true → _stopLocalProgress() → processVisual
```

**Agent 采集流程（原架构）：**
```
Agent → POST /acquire/start → 后端 acquiring=true
WebSocket 500ms 广播 → 前端检测 acquiring:true 变化 → 启动进度条
WebSocket 500ms 广播 → 前端检测 acquiring:false 变化 → 停止进度条 + processVisual
```

Agent 路径完全依赖 WebSocket 的 500ms 间隔广播来检测状态变化，存在时序不确定性。

### 迭代过程

#### 2.1 尝试：轮询替代 WebSocket 检测（有问题）
将采集状态检测从 WebSocket 移到 500ms 轮询（`setInterval` → `GET /api/detector/status`）。轮询使用 `_prevAcquiring` 变量追踪状态变化。

**新问题**：`_startLocalProgress()` 设置了 `progress.value.acquiring = true`，导致手动按钮的 `!progress.value.acquiring` 条件为 false，`_prevAcquiring` 永远不会变 true，完成检测永远不会触发。进度条永不停止。

**修复**：`_startLocalProgress()` 中同步设置 `_prevAcquiring = true`。

#### 2.2 尝试：setInterval 并发问题（有问题）
`setInterval` 不等前一个回调完成就发下一个。retry 循环（5 次×500ms=2.5s）期间多个回调并发执行，同时读写 `_prevAcquiring`，造成竞态。

**修复**：改为递归 `setTimeout`——在回调末尾才排下一个，保证串行。`_fetchVisualWithRetry` 改为 fire-and-forget（不 await），不阻塞轮询。

#### 2.3 尝试：WebSocket 推模型（后端立即推送）
在 `POST /acquire/start` 和 `/stop` 中加了 `_push_detector_status()`，不等 500ms 循环，立即推送 WebSocket。

#### 2.4 尝试：简化架构——WebSocket 直接控制进度条（图像恢复，进度条仍卡）
WebSocket handler 直接根据 `det.acquiring` 调 `_startLocalProgress()/_stopLocalProgress()`。轮询只负责 `processVisual` + `fetchHistory`。

**新问题**：`_startLocalProgress()` 只启动了 100ms timer，`progress.value.acquiring` 要 100ms 后才变 true。如果采集在这 100ms 内完成，WebSocket 收到 `acquiring:false` 时 `progress.value.acquiring` 仍为 false，`_stopLocalProgress()` 跳过，进度条永不停止。

**尝试修复**：在 `_startLocalProgress()` 开头同步设 `progress.value = { acquiring: true, percentage: 0 }`，消除 100ms 间隙。**但用户反馈手动采集图像又不显示了**（原因未完全确定，可能和删除 `_prevAcquiring = true` 有关）。

**回退**：删除同步 `progress.value` 设置，恢复 `_prevAcquiring = true`。

### 当前状态
- 手动采集：图像正常、进度条正常
- Agent 采集：图像能显示（`processVisual` 能触发），但**进度条可能仍卡住**（100ms 间隙问题未解决）

## 三、进度条回调复活问题（`clearInterval` 竞态）

### 根因
`clearInterval` 只阻止未来调度，浏览器已放入任务队列的回调仍会执行。`_stopLocalProgress()` 先 `clearInterval` 再设 `progress = { acquiring: false }`，之后队列中的回调执行，设回 `progress = { acquiring: true }`，进度条复活。

### 修复
添加 `_stopped` flag：
```typescript
let _stopped = false

// _startLocalProgress(): _stopped = false
// _stopLocalProgress(): _stopped = true
// timer 回调: if (_stopped) return
```

---

## 四、系统关机端口清不干净

### 根因
`system_shutdown` 用 `context.metadata["startup_pid"]` 来 `os.killpg`。metadata 在 OpenHarness 中每次 tool 调用都会重置（从 `context.tool_metadata` 展开副本），跨调用丢失 PID。

### 修复
PID 能杀则杀 + `fuser -k 8000/tcp` 和 `fuser -k 5173/tcp` 作为兜底。

---

## 五、其他修复

- **`vis_msg` UnboundLocalError**：初始化从 `if baseline` 分支内移到分支外
- **Agent 采完基线调 processVisual 加重试**：5 次×1s 延迟，失败报 "⚠️ 基线存储失败"
- **轮询排除 `acquiring` 字段**：防止轮询的 `Object.assign(status, s)` 覆盖 WebSocket 设置的 `status.acquiring`，干扰 WebSocket handler 的 `wasAcquiring` 变化检测（后期架构调整后此修复被替代）

## 七、架构教训

1. **前端手动操作是最好的参照物**——手动能工作、Agent 不能，差异一定在检测机制而非后端 API
2. **`setInterval` + async 回调 = 并发 bug**——递归 `setTimeout` 才能保证串行
3. **`clearInterval` 不可靠**——已入队的回调仍会执行，需要 flag 兜底
4. **同步 vs 异步间隙**——手动按钮的 `progress.acquiring = true` 是同步的，任何替代方案如果延迟了这步就会引入 bug
5. **OpenHarness 的 `context.metadata` 不跨 tool 调用共享**——每次 tool 调用创建新副本，状态必须持久化到磁盘或后端
