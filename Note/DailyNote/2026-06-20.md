## 一、设计回顾

### 1.1 Skill 深度分析

三个 Skill 的设计原理回顾：

| Skill | 角色 | 核心机制 |
|-------|------|---------|
| `experiment-control` | 行为约束层 | description 触发匹配、5 条元指令、工作流模板、B/C 状态机 |
| `safety-rules` | 免疫系统 | 14 条 IF-THEN 规则、隐式严重性分级、纵深防御 |
| `troubleshooting` | 免疫记忆 | 诊断策略优先、三级阶梯排查、Tool 引用而非命令 |

**核心设计原则**：Skill 描述"做什么"和"为什么"，Tool 负责"怎么做"。Skill 不包含 HTTP 调用。

### 1.2 Tool 设计原则

36 个 Tool 的通用 DNA：
- Input Model（Pydantic, 类型+范围校验）
- BaseTool 子类（name + description → Agent 看到的函数签名）
- execute() 三段式（调 API → 判成败 → 翻译中文）

关键设计概念：
- `context.metadata`：跨 Tool 共享状态（PID、文件路径、基线状态）
- 复合 vs 细粒度：认知原子性为封装标准，非操作步骤数
- 纵深防御：Skill 层警告 + Tool 层 Pydantic 硬校验 + 后端业务逻辑

## 二、环境与部署

### 2.1 conda + uv 双环境

OpenHarness (uv) 和后端 (conda slsdet9) 是两个独立进程，通过 HTTP 通信，不会冲突。

### 2.2 Linux 文件结构

```
~/JF_Agentic_System/
├── JF_Control_System/       ← 后端+前端
└── OpenHarness/             ← Agent 运行时
```

必须是兄弟目录——`system_tools.py` 通过相对路径 `../../../../../JF_Control_System` 定位后端。

### 2.3 运行方式

```bash
cd ~/JF_Agentic_System/OpenHarness
uv run oh
# 然后对 Agent 说："启动实验系统"
```

## 三、Bug 修复记录

### 3.1 plugin.json 空字段导致崩溃

**问题**：`"hooks_file": ""` → `Path / ""` = 目录 → 读目录当 JSON → IsADirectoryError

**修复**：去掉空的 `hooks_file` 和 `mcp_file` 字段（提交 `a7c433c`）

### 3.2 Chiller MODBUS 数值缩放

**问题**：水冷机和流量返回值需 ÷100 才是实际值

**修复位置**（提交 `8ed555c`）：
- `ChillerGetStatus.execute()`：`temp = data["temperature"] / 100.0`, `flow = data["flow_rate"] / 100.0`
- `ChillerWaitStable.execute()`：读取时 `/100`
- `DetectorRunAcquisition.execute()`：安全联锁温度比较时 `/100`

### 3.3 WebSocket 前后端协议不兼容

**问题**：后端发送 `{displacement, chiller, detector}`，`useDetector.ts` 期待 `{type: "status_update", data: {...}}`

**修复**（提交 `43f4ee2`）：`useDetector.ts` 的 `onMessage` 增加新格式优先判断：
```typescript
if (msg.detector) {  // 新后端格式
    Object.assign(status, msg.detector)
    // ...
    return
}
// 旧格式兼容保留在后面
```

### 3.4 安全联锁只在信号模式生效

**问题**：`if arguments.mode == "signal" and arguments.check_safety` — 基线采集不检查水冷

**修复**（提交 `9990316`）：去掉 `mode == "signal"` 条件，任何采集都检查

### 3.5 旧 Skill 未清除

**问题**：`.claude/skills/experiment-control/` 旧 curl 版与新 Plugin 版同名冲突

**修复**（提交 `5d1ed5c`）：`git rm -r` 删除旧 Skill

## 四、功能增强

### 4.1 默认配置路径 + 文件浏览

- 默认配置：`/home/jfdaq/JF500K/JF500K-shine.config`
- `LoadConfigInput.path` 和 `RunAcquisitionInput.config_path` 设为此默认值
- 新增 `detector_browse_files` Tool（封装 `GET /api/detector/browse`）

### 4.2 raw 文件追踪

`detector_run_acquisition` 采集完成后：
1. 读取 `fpath`/`fname` 参数
2. 构造 `{fpath}/{fname}_d0_f0_*.raw` 路径模式
3. 存入 `context.metadata["last_fpath"]` / `["last_fname"]`

`processing_analyze_acquisition` 自动从 metadata 获取路径，用户不需要手动传。

### 4.3 基线状态追踪

```python
# 基线采集后：
context.metadata["has_baseline"] = True

# 信号采集前：
if not context.metadata.get("has_baseline"):
    return error("请先执行一次基线采集")
```

### 4.4 探测器参数确认步骤

工作流增加 step 5-6：

```
5. detector_get_params → 展示全部 10 个参数
6. 询问用户是否修改：
   曝光时间、帧周期、帧数、偏置电压、时序模式、
   文件名、存储路径、读出速度、Power Chip、写入文件
   （用户已在指令中指定参数则跳过）
```

参数列表来源：`frontend/src/components/detector/ParamSettings.vue`

### 4.5 水冷优先规则强化

Skill 工作流顶部增加硬性声明：

> ⚠️ 强制规则：任何探测器操作之前，必须先完成水冷检查和温度稳定。不能跳过。
